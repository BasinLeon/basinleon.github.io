# Operator Library Architect (1000% Standard)

## When to Use
Use this skill for any work involving the @Library inside THE GAME or the user's personal operator practice:
- Adding new archiving hooks (Court readings, Sam & Ink captures, Codex applications, Book resolutions, etc.)
- Designing "Consult the Archive" mechanics and buffs
- Building Library snapshot or individual entry exports for X
- Evolving how the Library integrates with CodexHub, power-ups, and the movement
- Turning accumulated field notes into higher-order operator technology

This skill ensures the Library never becomes a dead archive — it stays a living, growing text.

## Core Mandate
The Library is the operator's personal Zohar. Every entry must be:
- Worth re-reading later
- Worth re-broadcasting on X
- Capable of producing real in-game or real-world effects when consulted

## The Process

### 1. Map the Transmission Type
For any new archiving need, define:
- **Type**: court-reading, sam-ink, codex-application, book-resolution, field-event, etc.
- **What gets captured**: The raw reading/text + minimal high-signal stats (never noise)
- **When it auto-archives**: Immediately after meaningful action, never on every micro-event
- **Re-broadcast value**: What makes this entry worth exporting again months later?

### 2. Design the Entry Structure
Keep entries lean but rich:
- id, type, text (the actual transmission/reading)
- keyStats (sparks, network, layers moved, powerup applied, etc.)
- timestamp
- Optional: linked powerup or context

Never store raw game state dumps. Store the *meaning*.

### 3. Build Consultation Mechanics
"Consult the Archive" should feel like a real operator practice:
- Random meaningful pull (weighted toward high-signal past entries)
- Small but real buff that matches the entry's energy (clarity from old high-clarity shots, network movement from past transmissions, etc.)
- Sometimes surfaces a forgotten pattern the operator needs to see again

### 4. X Export Design (Always)
Every Library-related feature must produce postable artifacts:
- Individual entry re-exports (the moment, preserved)
- Library snapshots (current state of the operator's text)
- "Best of" or thematic collections when the Library grows large

Use the established perfect X connection pattern (download + clipboard + compose with proper @CodexHub @Applications tags).

### 5. Integration with the Rest of the Movement
- New power-ups and applications should write to the Library by default
- Sam & Ink transmissions are first-class Library citizens
- The Library can be a source of new Codex fragments or power-up ideas over time
- Consider "Library as Codex material" — long-term patterns in the archive can become new power-ups

### 6. 1000% Quality Gate
Before shipping any Library change:
- Would an operator 6 months from now still care about this entry?
- Does re-exporting it produce something you'd actually post today?
- Does consulting it create a feeling of "the field remembers" rather than just a stat bump?
- Is the writing in the exact voice of the transmissions (precise, slightly cold, high-resolution, movement language)?

## Voice Rules
- Library entries and UI text must match the Court readings and Sam & Ink transmissions.
- No corporate "knowledge base" language. This is not Notion. This is the operator's accumulating text.
- When in doubt, make it rawer.

## Exit Criteria
The Library work is complete when:
- Meaningful actions automatically become consultable wisdom
- The operator feels the weight and value of their own history
- Exports feel like broadcasting pieces of their personal Zohar
- It creates emergent behavior (old entries inspiring new power-ups, patterns across time becoming visible)

This is how scattered actions become a coherent operator life.
