# Nexus Ghost Mirror Protocol

A "Ghost Mirror" is a private, client-specific dashboard or UI that proves the existence of a high-status system, bridging the "Logic Gap" before a contract is signed. It shifts the conversation from "I can build this for you" to "Here is the engine already built; we just need to turn it on."

## Core Principles

1. **Strategic Privacy**:
   - Absolutely NO indexing. Every ghost mirror MUST include `<meta name="robots" content="noindex, nofollow">`.
   - Hosted privately (e.g., via GitHub Pages at `basinleon.github.io` unlinked).

2. **Visual Aesthetics (Sovereign UI)**:
   - Use high-status, dark-mode styling (e.g., `#0A0A0A` backgrounds, `#FFFFFF` text, subtle neon accents like `#CCFF00` or `#FF3366`).
   - Employ "Terminal" or " Cockpit" aesthetics. Use mono fonts for data rails and sleek sans-serifs for headings.
   - Glassmorphism for overlays and panels.

3. **Data Representation (The Illusion of the Live Engine)**:
   - Do not use generic "Lorem Ipsum." Use highly specific, hyper-relevant data points for the prospect.
   - For example, if pitching Turner Construction, the mirror must show "Riverside Medical Center" as a live node.
   - Use dynamic-looking numbers (e.g., System Load, Recall Pct, Signal Heat).

## When to Execute
- When navigating a high-stakes deal (e.g., MTI, NBC/Jassi, Bounty.ai) where a standard PDF is too low-status.
- When you need to justify an anchor price (e.g., $2,000/mo) by visualizing the infrastructure.
- To serve as a visual companion to a "Diagnostic Call" or "Strategic Yield Brief."
