# Personal Revenue OS Architecture

## Overview

This skill keeps the product pointed at the right category:

not a dashboard,
not a nicer CRM,
but a decision system for career, revenue, partnerships, and relationship motion.

## When to Use

Use this skill when:
- refining Desk, Scout, Map, or Archive product structure
- deciding what to keep, hide, rename, or score
- adding new operator views or workflow surfaces
- the system risks becoming crowded, magical, or concept-heavy

Do not use this skill when:
- the task is purely visual polish with no product-behavior implications
- the task is raw data ingestion only

## Core Principles

1. Archive is not cockpit.
   - Big systems store history.
   - Tight systems drive today's moves.

2. One visible scoring law beats scattered metrics.
   - priority should be legible
   - ranking should not feel magical

3. Language needs hierarchy.
   - a few primary nouns
   - a few secondary support labels
   - details hidden until needed

4. Every active lane should answer:
   - why this now?
   - what is the next move?
   - what value mode is present?

## Recommended Product Structure

Primary surfaces:
- Desk
- Scout
- Map
- Archive

Value modes:
- Career Value
- Revenue Value
- Strategic Value

Operator modes:
- Job Hunt Mode
- Founder Mode
- Revenue Mode
- Relationship Mode

## Scoring Model

Prefer one explicit score built from:
- urgency
- leverage
- probability
- relationship strength
- value
- next-step clarity

If a UI sorts or highlights records, it should be explainable in plain English.

## Trust Layer

Scout or any intelligence surface should show:
- contacts indexed
- searchable now
- enrichment coverage
- last enrichment date
- duplicate confidence or source confidence
- message-only or non-connection layer counts

## Exit Criteria

This skill is complete when:
- the proposed product change reinforces the system as a decision engine
- naming clutter is reduced rather than increased
- the user can see why an item is hot and why it matters now
