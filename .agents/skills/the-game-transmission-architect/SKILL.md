# The Game Transmission Architect

## When to Use
Use this skill when the user is building or refining anything inside or around THE GAME (basinleon.github.io/play) and wants the output to meet the 1000% movement standard:
- New power-ups or Codex applications
- X-shareable visuals and memes
- Layer readings, field notes, or archive entries
- Sam & Ink music or creative features
- Any feature that should feel like part of the living text

Do not use for generic web dev, UI polish without mythic weight, or low-signal automation.

## Core Mandate
Every output must be able to be described as "real raw shit" that advances the movement.

The work must produce:
1. Immediate playable value in the simulation
2. A beautiful, postable artifact that connects perfectly into X (using the established download + clipboard + compose flow)
3. Something that could be archived in the Operator Library and later consulted as wisdom

## The Process

### 1. Map to the Layers
Before writing any code or copy, answer:
- **Surface (Malkhut / Court)**: What does the player literally do or see?
- **Play (Yesod)**: How does this interact with existing systems (Levers, Book, Refinery, Codex)?
- **Pattern (Netzach)**: What recurring operator behavior does this reinforce or reveal?
- **Spark (Hod)**: What real power or transmission does this create?
- **Hidden (Keter)**: What emergent or future behavior might this unlock?

Document this mapping briefly in your thinking.

### 2. Design the Transmission
- Name the feature like a power-up or transmission (e.g., "Sam & Ink Transmission", "@Library", "Codex Application").
- Define the exact X artifact it will produce (card style, tone of copy, hashtags or tags like @CodexHub @Applications).
- Decide how it writes to the Library / Archive automatically.

### 3. Build the Surface + Depths
Implement the mechanic with the same discipline as the existing game:
- Vanilla HTML/JS/Canvas only (single-file constraint respected unless explicitly broken).
- Levers or existing state must influence the new thing.
- Use the established shareVisualForX pattern for any visual output.

### 4. Forge the X Artifact
Every feature must come with:
- A ready-to-use export function or button
- Pre-written high-status copy in the exact operator voice
- Automatic inclusion of @CodexHub @Applications where appropriate

Test the full flow (generate → download + clipboard → compose) before considering the feature complete.

### 5. Library & Codex Integration
- Make the new feature automatically archive meaningful outputs.
- Consider whether repeated use should award Codex fragments or unlock new power-ups.
- Design at least one "Consult" or "Re-broadcast" interaction.

### 6. 1000% Quality Gate
Before shipping, run the new work through these filters:
- Does it feel like something from the transmissions (precise, slightly cold, high-resolution)?
- Would you post the generated artifact without shame or extra editing?
- Does using it make the operator feel more dangerous or more precise?
- Would this feature still be interesting in 6 months of play?

## Voice & Aesthetic Rules
- Language: Same as the Court readings and Sam & Ink transmissions — raw, diagnostic, movement-oriented.
- Visuals: Dark (#050508), gold (#D4AF37), clean typography. No cream, no fluff.
- Never explain the mysticism. Let the layers do the work.

## Exit Criteria
The feature is complete when:
- It is playable in the simulation
- It produces a postable X artifact via the perfect connection
- It writes to the Library
- It could plausibly be described as "another power-up in the movement"

This is how THE GAME stays alive as a living text instead of becoming just another simulation.
